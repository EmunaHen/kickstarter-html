Story

360Crafter empowers creators to build beautiful 360° virtual tours without compromise. Self-host for complete data control, customize every detail, and create immersive experiences that truly belong to you.

360Crafter Features (v1)

Create a workspace to work as a team

Create a workspace, choose to connect it to your own s3 storage bucket. Then you can invite your teammates and clients to the workspace, so that you can collaborate on projects while keeping work in sync using your own s3 storage. No more exporting backup projects that need to opened by another user who imports files from a different folder creating missing files and all those type of headaches. Everyone works on the same files. Hopefully real-time.

Connect your own s3 storage bucket

Creating a workspace you can choose to have the files in the bring your own s3 storage bucket. Bringing your own s3 storage bucket you will need to provide a few things such as api keys, url and some other details.

Hotspot Styling

Skin Design

In the skin you can design components on top of your 360° viewer to create something like a left side menu, mobile responsive with breakpoints. Text, image, video-url's, embedding websites like google maps location. Adding actions to the skin components when the user clicks on any of them.

Hotspot minimization

There's panorama settings to minimize all hotspots in a panorama, which make it feel like the application that lets you view streets in 360.

Hotspot tooltips

Setting to show the hotspot names as a tooltip on rollover of a hotspot.

Interactivity by adding Actions

And in order to make it all interactive, you can appy actions to both skin and hotspots as your triggers. Actions to show or hide hotspots, show or hide skin components, action to show a temporarily or fixed notification to inform the user of something, open url and copy deeplink to navigate to the exact camera view the user was looking at.

Export project

Once you're finished working on the project, you can start exporting the 360° virtual tour. Both locally to your computer as well as directly to your s3 storage.

Having it locally on your computer might serve as a backup to archive which can be imported later again into 360Crafter when you need to.

Quality of Life design

Software should be easy to use, yet bring you the flexibility to customize. The tours that you will be creating yourself and showing to your customers should be easy to use, as well as the software itself that you're building it with.

Everything in 360Crafter is within the details making it easy. Take for example adding a new action to a hotspot. Notice how a "1" appears next to the Actions button, as well as a "⚡️" icon on the hotspot itself on the left side panel. It's these small details that make you understand the scene instantly without checking each hotspot one by one to see where you might have added that action that's messing something up.

And there's many more details in this. The "👀" icon, the 6 dots drag icon to change order of both panoramas as well as hotspots. The consistent use of icons across the application. When deleting an action, hotspot, panorama, project or workspace that it first asks you if you are sure to delete it. But also the ability to ⌘ command or control click the trash icon to skip this question and just delete it instantly. Making sure command/control "Z" actually just works undoing an action. It sounds basic, but these basic things just need to work as you might expect of a modern application.

Instead of having multiple actions to Show/hide hotspot, show/hide components, open pdf popup, open image popup, it's all combined into one single action called "Edit Visibility". And in there you can choose whether you want to edit Skin visibility of Hotspot Visibility.

Once added the action, you will see the action appear in "Applied actions". And you see the icon that represents visibility, the title that tells you it's visibility related to the Skin and a subtitle that shows you exactly what you are showing or hiding, so that you do not need to edit the action in order to see what's on there. And of course a delete icon to delete the action.

Enshittification

Software can look great, have amazing features, work really well, deliver a ton of value. Start for free, built community, listen carefully to everyone and grow market share as the software gets develops. Until it dominates the market. Then it can start exploitation. Make the app worse, add in ads, spy more, move features to higher price tiers, increase pricing, let AI train on customer's software by forcing clients to click accept or else they don't have access anymore. Most of this enshittification comes from the fact that most software is proprietary closed source software. Meaning that everyone got fully dependent on it and it's made very difficult to say "No thank you" because you can't just take the latest version that you liked and use that. Everyone's supposed to either accept or delete their account.

Times when it went wrong with closed source software

There's been lot of times where companies are taken over and have disappeared. Think of GoPro killing Kolor. What if pricing goes up and the privacy policy changes to let the company that you disagree with like Adobe did a year ago or more recently WeTransfer? Or if you upload your data to a company like Matterport providing them with an unrestricted, irrevocable, perpetual, transferable, sublicensable, worldwide, royalty-free license to use, reproduce, display, publicly perform, transmit and distribute any such material you submit, without any payment or accounting to you or others? All of this is no good...

To learn more about enshittification, I'd recommend watching Techlore's video on it and videos from Louis Rossman.

What's the remedy against enshittification?

The more you dive into this subject, the more you will learn that when a company offers you the ability to fully self-host the application on your own server, you're getting in much better hands. Even if you don't know how to do it, their cloud subscription has much lower chances of getting enshittificated. Why? Because if the software gets worse, clients start moving away from their cloud and self-host the application on their own. When you're able to view the source code of the application, you're able to check what happens in there. Even if you don't know how to code, you can ask an ai to check things for you. Open source and source available might not be the full remedy against enshittification, but it's a huge step in the right direction for sure.

How is 360Crafter going to be licensed?

That's exactly why 360Crafter is going to be licensed as source-available following the fair-code principles similar to how n8n does this with a Sustainable Use License.

You can get a full copy of all its source code and run it on your own server. You can use the source-code as it is, completely for free, make money with it creating and selling virtual tours as you are supposed to. But changing the source code and making money with it as if it is your software running it on your own server requires a different commercial license. That's the idea of a Sustainable Use License.

Having the source code available allows anyone to always downgrade or upgrade to their own liking, review the code as they want, make sure all the data is on their server to have it GDPR compliant. And all of this together makes sure there is no vendor lock-in. That's freedom that very few SaaS companies offer anyone nowadays. And that is exactly how 360Crafter plans to create real longterm trust in the 360° virtual tour industry.

How is 360Crafter created & maintained?

For a very long time I thought about creating my own 360 Virtual Tour software and the cost I would have to hire a team of developers for at least a year and the huge investment I would need to pitch this idea to investors. Maybe €500.000? Having investors on board would mean I need to find investors that understand and share the same values of open-source and fair-code which is quite a large challenge already in itself. Beside that, investors generally speaking intent to motivate enshittification in order to make more money. So I discontinued the idea and moved on with other things quite some time ago in the past.

Until I started prompting in Lovable and Cursor 25th of June. I figured that a lot of these things I could prompt my way through and do it myself. So far I've spent over 100 tracked hours on 360Crafter. I've figured out how to ask the questions, solve complex bugs and use my imagination to make it in the way I think makes sense.

I'm aware there's vulnerabilities being a so-called vibe-coder. That there's things on security to double check with developers that understand code much better than I do. Allthough I ask the ai tools to be secure and not save API keys on the server whenever possible to avoid that using zero-knowledge strategies. But it does need to be checked anyway. That's going to be part of the cost creating this project, but it's going to cost much less than having a team of developers working full time on a project that hasn't sold itself yet on the market and having investors on board with different values. So I'm quite happy I need a much lower investment than I initially thought I would need.

Vision of 360Crafter (v2)

As soon as v1 is finished, 360Crafter can start working on the extra awesome features of v2 after December this year.

Design tokens

Nothing more annoying than changing the background color on 100 hotspots manually. Instead of using css classes and combo classes, 360Crafter will be using Design Tokens which brings more flexibility to design with ease. It's a modern industry web standard being used by Webstudio, Tailwind & Penpot.

Static CMS

From experience building skincms connecting Airtable with 3DVista, I can tell that external systems to manage data can be both a blessing as well as a curse. That's why I want to start creating a CMS that's based on formats like CSV or JSON that can be exported from many applications, and then imported into the 360Crafter. The idea is that when the user exports the virtual tour, the data from these files comes static as well with this extra file in the exportfolder. Making it a single package to deliver to the client and still keeping the tour relatively easy to manage as there's no need to duplicate a popup many times.

Dynamic CMS

Static CMS has it's benefits being a single package to deliver, but the downfall with it is that it's not in sync with actual data being used by whatever system the client is using. With a dynamic CMS, the client can have it's own systems to update content and it's being reflected live on the exported tours. Whether that is Airtable, Baserow, Strapi, Wordpress, Ghost, Craft, Notion or any other CMS system is up to you. Users can connect a skin popup easily to a database. Or add dynamic tooltips to hotspots, edit visibility state and many more interesting things.

Publish tour to 360Crafter CDN

A CDN stands for Content Delivery Network which generally has many servers all around the world. Let's say you are located in Amsterdam and hosting your files there. If someone from New York visits the virtual tour, it needs to travel a long distance virtually which takes more seconds to reach the server and load in the data.
With a CDN there's servers all over the world and the user in New York will be automatically connected with the server closest to his location, improving loading time tremendeously.

Long-lasting design

The project will be publicly available on a Github or Gitlab page with version control. Each version is saved there publicly. If for whatever reason I would quit on 360Crafter and stop managing the cloudversion of my product, there's options to selfhost the application. Beside that even people using the cloudversion, can connect their own s3 storage so that 360Crafter doesn't save projectdata and exportdata into their own cloud. If the 360Crafter application server would go down, people still have access to their own data who use their own s3 storage and can reconnect it with hosting the application on their own server with the publicly available source code if needed.

Reusability and recyclability

I know this question is more for creatives that use kickstarter to create a fysical product, but I still want to answer this question as a software developer. My "materials and packaging" are mainly the code dependencies which are open source and permissive licenses like Three.js being one of them. My intent is to have a percentage of my income donated to the maintainers of these dependencies, so that they can keep working on it and pay their bills.

Sustainable Distribution

I'm planning to host the 360Crafter application using a Hetzner server which have written this on sustainability: https://www.hetzner.com/unternehmen/nachhaltigkeit/

Something else

I've given it a lot of thought whether I should go for an open-source license myself, whether that should be permissive or copyleft. And whether that is sustainable. There's a lot of debate whether giving away everything completely for free and letting anyone copy your code and reselling that is fair or not. Fair-Code is not open-source according to OSI (https://opensource.org/), but I think it does solve the commercial aspect to have commercial limitations on using the source code while following a lot of open-source values such as providing privacy with the ability to host it where people want to, insights in the code and removing artificial limitations in order to sell something. Combining the technicalities with the legal aspects to have a middle ground for financial success while still being very open about the whole thing.
