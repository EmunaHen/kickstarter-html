What is SensorNode?

SensorNode is a simple credit-card sized PC board with a Raspberry Pi Pico W installed on it. The Pico is loaded with software that enables many common home-automation sensors to be interfaced to Home Assistant, one of the most popular open-source home automation software solutions available. All you do is attach sensors to it, configure it through the built-in Wifi Access Point, and run. Although all the code is Open-Sourced, you don't need to write code to use SensorNode.

Key Features of Every SensorNode

  • Every SensorNode can be configured to interface Home Assistant to as many as Four concurrently reporting sensors:
    • ONE Dry Contact External Sensor - for Push Buttons, Switches, Door/Window sensors, Float Sensors, Vibration/Tilt Sensors, etc.
    • ONE of the following additional external sensors: Motion, Temperature, or Moisture (STRETCH GOAL: Human Presence Detect Sensor support - see below)
    • I2C Interface to add TWO additional sensors (currently only supports AHT20 internal temperature & humidity sensors) for a total of 4 concurrent sensors!

  • Easily configurable with your smartphone or computer - NO App, NO Cloud, No Subscriptions - ALL Data is completely LOCAL!

Detailed Feature Descriptions

S1 - Dry Contact Sensor

Connector S1 on SensorNode is a 2-conductor sensor connector which will accommodate any 'Dry Contact' sensor. Dry Contact is a fancy way of saying this sensor is passive, that is it provides NO VOLTAGE; it simply opens or closes a contact (like a dumb switch). In fact, you can connect any dumb switch (or just short two wires together) to this connector and it will work (as long as you enable it in the settings).

S2 - Configurable Sensor (Motion, Temperature, or Moisture)

Connector S2 on SensorNode is a 3-conductor sensor connector, meant to attach sensors with POWER (5V), GROUND, and DATA lines to SensorNode. This connector is designed to accept ONE of three types of Sensors:

  • Motion Sensor

Most cheap motion sensors are all the same, with power, ground, and data leads. Typically, the data pin simply goes HIGH or LOW if it sees motion. SensorNode uses a hardware latch to capture this signal to ensure it is not missed while the processor is busy doing other things. In the parts list, I will specify motion sensors that can easily be purchased that work with SensorNode, or you can purchase the add-on sensor from me.

  • Temperature Sensor

SensorNode was designed to interface S2 to a DS18B20 'Dallas Semi' type of temperature sensor which uses a very common digital '1-wire' interface (in actuality, the interface is THREE wires: POWER (5V), GROUND, and DATA. The DATA line implements a serial protocol to transfer digital data, thus the '1-wire' nomenclature). This type of temperature sensor is VERY common, but if you want to get one that I've tested and will be sure to work with SensorNode, you can purchase the add-on from me.

  • Moisture Sensor

I added this because I wanted a simple moisture sensor to sense leaks. Adafruit offers a very simple moisture sensor which uses - you guessed it - THREE signals: POWER(5V), GROUND, and a DATA signal. SensorNode will latch this data signal as well, to ensure the Pico does not miss it if the moisture sensing is intermittent.

--> THE TRICK to using any ONE of these THREE TYPES of Sensors lies in JUMPER J3 on the SensorNode PC Board. This jumper MUST be placed in the proper position to tell SensorNode which type of Sensor is connected on the S2 connector. In addition, the proper Sensor TYPE must be specified in the configuration settings (and the sensor must be enabled) before the Sensor will work.

NOTE:  You can buy these and the other sensors direct from the sources I will list in the instructions shipped with your rewards - you don't have to buy them from me. In fact, you can probably get them cheaper than you can from me, but if you want to make sure you get the exact sensor type I've personally tested, you can get the add-ons from me.

Home Assistant Interface

SensorNode talks to Home Assistant via the Home Assistant HTTP Sensor Interface. This is a very simple interface in which the device uses HTTP POST commands to make sensor reports to Home Assistant. For this to work, you must first specify the following in the SensorNode Configuration Settings (through the WiFi Access Point):

  • Home Assistant URI - this is the network path to your local Home Assistant install. By default, SensorNode is set to the Home Assistant default path: "http://homeassistant.local:8123" but you can change it to anything you want. Note: SensorNode MUST be on the same network as Home Assistant to work.
  • Home Assistant 'long-lived' access token. This will authenticate SensorNode to Home Assistant. You need to go into Home Assistant settings, obtain one of these tokens, and then paste it into your SensorNode Settings.

Setup and Configuration with a Smartphone - No App Needed!

One of the reasons SensorNode is so easy to set up and use is the fact that you do not need an app or any cloud service to use SensorNode. Just plug it in and turn it on. The first time SensorNode is turned on, it will attempt to join the local WiFi, but if it does not have correct credentials (or for some other reason this fails), SensorNode will immediately create a WiFi Access Point (AP) that you can attach to with any smartphone, computer, etc.

After you attach to SensorNode's AP, simply open a browser and point it to 192.168.4.1 and the SensorNode Configuration Server will open. From there you can set up SensorNode with WiFi credentials, Home Assistant info, and enable all desired sensors. You can even give SensorNode and/or the sensors custom names so you can more easily identify them on your Home Assistant dashboards.

If you ever want to change any of the SensorNode's settings, there is a switch on the board to force the unit into AP mode. Simply turn it off, throw the switch to 'AP', then turn it back on. SensorNode will go into AP mode and you can modify any settings you wish. After you are done, throw the switch back to 'Normal' mode, and hit 'Submit' on the Configuration Home Web Page, and SensorNode will restart into Normal mode.

I2C Interface

One of the advantages of using the Pico is that is has a built-in I2C interface. There are literally dozens of I2C devices out there, but for now I've only implemented one - the AHT20 combo temperature and humidity sensor. This is primarily intended to be an 'internal' sensor close to SensorNode, but for those who want it, you can either buy the AHT20 directly from Adafruit or you can get the add-on from me. All you have to do is plug it in and turn SensorNode on. It's that easy. Also, I've put both a 'STEMMA' connector and a standard header on the SensorNode board - either will work.

Easy to build yourself

I designed the SensorNode PC Board to make it easy for me to build for myself. It uses simple, cheap components. All the parts are through-hole (except the STEMMA I2C connector which you don't absolutely need) so if you know how to solder it shouldn't take you long to build your own SensorNode. Of course, you can purchase the already-assembled versions, but the point is you don't have to.

The Back Story

My name is Tom Berarducci - I'm a retired electrical engineer who used to build things for big companies. Now I build things for myself. Recently, I had to replace my aging home automation system with something new. I looked around and decided on Home Assistant. I started adding devices and quickly found that, although there are a TON of devices available for Home Assistant, I could leverage my experience in IOT (Internet of Things) work to add my own unique take on sensors to not only save money, but increase simplicity, robustness, and flexibility to my Home Assistant installation. Then, I considered that others might benefit from this work.

Thus, SensorNode was born.

I chose things mainly based upon simplicity. I chose the Raspberry Pi Pico W as the main brain of the system, because it's cheap ($6-7), available, can be programmed in Python (I used CircuitPython), and has tons of General Purpose Input Outputs (GPIOs).

I chose AC power so I wouldn't have to bother with replacing batteries (see below). I chose the most common sensors I would want. I chose the Home Assistant HTTP interface for the SensorNode and all it's sensors cause it's simple and easy.

You can dynamically set up the SensorNode through the built-in WiFi Access Point. It can be set up for a Dry Contact Sensor on connector S1 and ONE of Motion, Temperature, OR Moisture Sensors on connector S2, along with optional I2C temp/humidity sensors. That gives you a possible 4 concurrent sensors which is just about all the Raspberry Pi Pico can handle.

These are my choices, for me. If you agree with them, AND if you are a person experienced in DIY, soldering, tinkering, power systems, etc., consider backing this project so that others can benefit from my work. Thanks.

Why?

Why would I do this? Well, I had several reasons. If you agree with them, you might also want your own SensorNode. Read on.

First of all - I'm not in this to make money. My main goal is to let people who are Home Automation Geeks like me enjoy using SensorNode. Of course, I'm not planning on losing money either, so that's why I need to charge what I'm charging. I don't have a manufacturing arm in China and am not planning on mass production. That's why I'm selling bare boards and open-sourcing the code and 3D print files.

The main reason I came up with SensorNode was, as I was setting up my Home Automation system, I had the need for several Dry Contact switches (garage doors, fountains, push buttons, etc.) and found that there were few options available. The ones that were available were expensive and battery-powered. Neither of which I wanted.

So, as any good engineer would do - I built my own.

But of course, as I'm designing it, I realized I could add additional sensors at very little cost. It was frustrating that a simple dry contact switch (that works with Home Assistant) costs $20-40, and were all battery-powered. Also, I figured out that it was possible to design a circuit that could accommodate multiple sensors at nearly the same cost. Now, you don't always need multiple sensors at the same location, but if you do, SensorNode is one of the cheapest ways to provide them.

But which sensors? Well, I quickly realized I couldn't provide ALL common sensors at the same time, the little Raspberry Pi Pico micro controller simply didn't have the power to do that and also provide the responsivity I was looking for. So I had to choose.

But I also didn't want a bunch of 'one-off' circuits all over my house. Too many unique boards, all requiring unique software, that had to be maintained. So I came up with ONE circuit, ONE board, and ONE software image that provided all the common sensor interfaces that I wanted - dry contact, motion, temperature, and moisture - all in ONE.

Plus, I threw in an I2C interface 'just in case' . Currently, this interface only supports ONE device - the AHT20 Temperature/Humidity sensor (I get mine from Adafruit). But, since the code is open-source you could add what you need.

Why AC Powered?

Most battery-powered devices claim 'up to one year battery life' and that's OK I guess, but that's not the whole story. The reason companies create battery powered devices is mainly for THEM, not you. It's easier. It avoids regulation and certification. They don't need to include a costly UL-Listed Wall Wart with their product. It pushes the burden of powering the device off on the user - YOU. It also makes the device safer - there's a reason children's toys are all battery-powered. 

But batteries leak and corrode. They create toxic waste. They should only be used if absolutely necessary. If you have unique needs like portability or locations which don't have accessible power, you need batteries. But home sensors rarely require these things. So I try to avoid them. Why? Mainly because I'm lazy - I don't want to be constantly replacing batteries. You see, one device needing battery replacement every 12 months is not bad. But most self-respecting home automation junkies have WAY MORE than one device in their systems. My system currently has over 80 devices. How many are battery-powered? Three. You see, one device needing battery replacement every 12 months is manageable, but if you had 80 battery-powered devices, you'd be replacing batteries, on average, every 4.5 days! Not how I plan to spend my Golden Years...

Current Status

The design is complete. I have over 20 units up and running around my house, several of them for many months non-stop. Some for over 1 year. The hardware is very simple so I'm confident it will be robust. 

The software, however, is kind of complex. Since the Pico is running CircuitPython, memory usage is a constant concern. I had to be creative about loading things to make sure I had enough memory in all of the various modes of operation. Because of this, the code is broken up into modules and only loaded when needed to make utilization as efficient as possible. To date, the code seems to work well and appears robust and fault-tolerant.

A Note about Security

Right now SensorNode uses HTTP for device-server communication exclusively. I have not implemented HTTPS for encrypted communication. To be honest, I'm not sure I will have the time, code space, or processing power to make that happen with this project. Technically this is a 'security hole', but honestly I'm not sure there is much risk for this type of device. SensorNode is meant to be installed locally, behind your existing Firewall, and it is ONLY meant to communicate with your install of Home Assistant on the same local network (you can set the URI in the settings). Also, SensorNode is a 'One-Way' device - that is, it is not meant to be talked to. There is no machine-machine input interface of any kind. Yes, it does create a WiFi Access Point for configuration, but this requires someone to gain physical access to put SensorNode into AP mode. So, it would be very difficult if not impossible for a Bad Actor to hack into a SensorNode. Someone could, theoretically, snoop the local network to see what SensorNode is transmitting, but they would have to gain access to the network by some other means. 

I would prefer to implement HTTPS communication for SensorNode, but I'm just not sure I can do it a this time. However, since the code will be open-sourced, perhaps someone else will do it.
